# AB Operations — 設計合意書

作成日: 2026-07-12(グリルセッションでの合意内容)
改訂: 2026-07-12(プロUI/UXデザイナー視点+シニアエンジニア視点のレビューを反映)

## 目的

Google カレンダー・Google Tasks・Gmail を1画面に集約した自分専用ダッシュボード。

- 最大の狙いは **「タスクが増えても予定が埋もれない」分離表示**(Google カレンダーではタスクと予定が同じ場所に表示され、タスクが多いと予定が見えなくなる不満の解消)
- **「1つのURLを開けば全部済む」司令塔化**(複数アプリを見に行く億劫さ・見落としの解消)
- 将来ここに色々な機能を統合していく土台とする

## 対応デバイス

自宅 Mac / 会社 Windows / Android / iPhone — すべてブラウザで同一 URL を開くだけ。
スマホは「ホーム画面に追加」でアプリ風に起動(PWA)。
**⚠ iOS(iPhone)のホーム画面起動では Google ログインが動かない事例が報告されており、実装のごく初期に iPhone 実機で検証する**(動かない場合、iPhone のみ Safari のブックマーク運用とする。Android は問題なし)。

## アーキテクチャ

| 項目 | 決定 |
|---|---|
| 方式 | 静的Webアプリ(サーバーなし)。各端末のブラウザが Google API から直接データ取得 |
| ホスティング | GitHub Pages(リポジトリ名 `ABOperations`、コードは公開されるが秘密情報は含めない。API キーは作らない・置かない — トークン付き fetch で直接呼べば不要) |
| アプリ表示名 | 「AB Operations」(画面タイトル・ホーム画面アイコン名) |
| 技術 | React + Vite + TypeScript。GitHub へのプッシュで GitHub Actions が自動デプロイ。**単一画面のためルーティング(URL によるページ切り替え)は導入しない**(設定はモーダルで表示。GitHub Pages の SPA 問題を丸ごと回避) |
| 認証 | Google Identity Services(クライアントサイド OAuth)。Google Cloud プロジェクトは**テストモード**で自分専用運用 |
| 設定保存 | 端末ごと(ブラウザの localStorage、端末間同期なし)。会社PCだけ Gmail オフ等、端末別設定がむしろ都合良い |

### 再ログインの現実的な頻度(レビューで判明した重要事実)

当初「まれに再ログイン1クリック」と見積もっていたが、実態は以下:

- **アクセストークン(API を呼ぶための一時的な通行証)は1時間で失効**し、GIS には画面に何も出さない自動延命の仕組みがない。→ 開きっぱなしで使う場合、**約1時間ごとに「再接続」ボタンを1クリック**(クリック起点ならポップアップは一瞬で閉じる)
- **テストモードでは同意自体が7日で失効**する(Google 公式仕様)。→ **各端末で週1回、未確認アプリ警告付きのフル同意画面をくぐり直す**
- 対応方針: (1) 401 エラー(認証切れ)検知時はパネルの表示データを残したまま「再接続」ボタンを目立つ位置に1つだけ出す UX を最初から組み込む。(2) **第1週の運用でこの頻度が苦痛かを実測**し、苦痛なら対策(TODO #8: 極小中継サーバー or 本番公開)を検討する
- **段階的認可(incremental authorization)を採用**: 初回は カレンダー+Tasks のみ同意し、**Gmail の権限はパネルを有効化した端末でだけ追加同意**する。会社PCのトークンは最初から Gmail 権限を持たず、「会社のポリシーを回避しない」方針により忠実になる
- アクセストークンは localStorage に永続保存せず**メモリ保持を基本**とする(XSS 対策。1時間で切れるので永続化の利点も小さい)

## レイアウト

- **PC(幅 1100px〜)**: 3カラム(予定 4 : タスク 4 : Gmail 2)
- **中間幅(700〜1100px、会社PCで画面半分にした時など)**: 2カラム(予定+タスク)、Gmail は折りたたみ
- **スマホ(〜700px)**: 画面下部タブ切り替え(予定/タスク/メール)
  - **初期表示タブは「予定」固定**
  - **タブにバッジ(件数表示)を付ける**: タスクタブに期限切れ件数(赤)、メールタブに未読件数。見ていないタブの異常に気づけるようにする(PC の3カラムで解決した「埋もれ」問題をスマホで再発させない)
- テーマ: OS のライト/ダークに自動追従+手動切替
- パネルは独立部品(コンポーネント)として実装し、将来のレイアウト切替・機能追加の土台にする
- レイアウト切替機能自体は後回し(まず3カラム型を使ってから判断)
- 設定画面は最初から用意(モーダル表示。Gmail オン/オフ、カレンダー選択などで必要)
- Gmail 読み込みオフの端末では、Gmail パネルを**細い縦帯に折りたたみ**、空いた幅を予定・タスクに再配分する(クリックで案内を再表示可能。機能差はない)
- キーボード操作: 最低限「Tab キーで項目を移動でき、フォーカス(選択中の枠)が見える」ことを守る。ショートカットキーは将来課題(TODO #7)

## 画面状態の設計(正常時以外の全状態)

ダッシュボードの品質は例外状態で決まるため、以下を最初から実装する:

| 状態 | 表示 |
|---|---|
| 未ログイン(初回) | パネルを出さず、アプリ名+「Google でログイン」ボタン1つだけのウェルカム画面 |
| セッション切れ | パネルは最後のデータを表示したまま、画面上部に「接続が切れました[再接続]」バナーを**1本だけ**(パネルごとに3つエラーを出さない) |
| 権限の一部拒否 | 該当パネル内に「◯◯の権限が許可されていません[再設定]」。他パネルは正常動作 |
| 空状態 | ポジティブな文言(予定「今後7日間は予定なし」/タスク「期限切れなし。すべて順調」/メール「未読メールはありません」)。真っ白なパネルはバグと見分けがつかないため必須 |
| 初回読み込み中 | スケルトンスクリーン(内容の形だけ灰色で見せる読み込み表示) |
| 更新失敗 | パネルのデータは消さず「最終更新 ◯分前・更新に失敗しました[再試行]」を小さく表示 |
| オフライン | 初期バージョンでは対象外(Service Worker は導入しない。TODO #6) |

## 予定パネル

- 表示: 今日から7日間+「さらに7日読み込む」ボタン
- **「今日」の強調**: 見出しを色・サイズで明確に区別。今日の終了済み予定はグレーアウトし、次の予定を立たせる
- 終日予定は各日の先頭にまとめて表示。複数日にまたがる予定は該当する各日に「(2/3日目)」のように表示
- カレンダーごとに色分け(+詳細表示にはカレンダー名をテキストでも出す — 色だけに頼らない)。設定でカレンダー単位の表示オン/オフ(初期値: 全オン)
- 操作: 単発予定の**作成・編集・削除**をフル対応
- 繰り返し予定:
  - 作成は**プリセット方式**(なし/毎日/毎週◯曜日/毎月◯日/毎年/平日)
  - 既存の繰り返し予定は**「この回だけ」の変更・削除**に対応
  - 繰り返しルール自体の変更(毎週→隔週など)のみ Google カレンダー本家で行う
- ゲスト招待・会議室・Meet 連携は対象外

## タスクパネル

- 表示: 全リスト混合で「⚠期限切れ(赤)/今日/今後/期限なし」の4グループ順
- 各タスクにリスト名の小さなラベルを表示
- 設定でリスト単位の表示オン/オフ
- 完了済みは非表示+パネル上部に「今日 ✓N件完了」を小さく表示(タップで展開、**そこから完了の取り消し=再オープンも可能**)
- 操作の3層配置(1行に5操作を詰め込まない):
  1. **常時表示**: 完了チェックのみ(タップ領域は最低44×44px。行タップ=詳細、チェックボックス=完了、と役割分離)
  2. **行タップで開く詳細**: 明日へ・来週へ・編集
  3. **詳細内のメニュー**: 削除(+確認ダイアログ)
- **完了操作には Undo(元に戻す)**: 完了時に画面下部へ「◯◯を完了にしました[元に戻す]」を5秒表示(スナックバー)。誤タップによる「タスクの静かな消失」を防ぐ
- クイック追加: 設定に「クイック追加のデフォルトリスト」を持つ。追加欄の脇に現在の追加先リスト名を小さく表示、タップで一時変更可。期限の初期値は「なし」+「今日」ワンタップ指定

## Gmail パネル

- 表示: 受信トレイの**未読のみ**・新しい順30件+追加読み込み(設定で「すべて表示」に切替可)。リストは差出人+件名の1〜2行表示
- **本文表示はモーダル**(画面中央に重ねて出す窓)で、カラム幅と無関係に十分な幅を確保する(幅1/5のパネル内に本文は物理的に入らない)
- 操作: **既読にする/アーカイブ/削除**
  - **「削除」はゴミ箱への移動**として実装(完全削除は最強権限が必要になるうえ取り返しがつかない。ゴミ箱なら30日間復元可能)
  - アーカイブ・削除には確認ダイアログではなく **Undo スナックバー**(Gmail 本家と同じ作法)。リスト上の一等地にはアーカイブ、削除は本文モーダル内に置く
- セキュリティ(メールは外部から任意の HTML を送り込める唯一の入口のため):
  - HTML 本文は **DOMPurify(定番のサニタイズライブラリ)で浄化+sandbox 属性付き iframe(隔離枠)で表示**の二重防御
  - **リモート画像はデフォルトでブロック**(開封追跡対策+サーバーレスでは自分の IP が送信者に見えるため)。「画像を表示」ボタンで明示解除
- 返信・新規作成は実装しない(連絡用途に使っていないため。将来必要なら追加)
- **会社PCでは Gmail の権限自体を同意しない**(段階的認可)。パネルは細帯に折りたたまれ、「この端末では読み込みオフ」と表示。機能差はなく、いつでも設定から有効化可能

## データ更新

- **復帰時更新を主、定期更新を従**とする: タブが前面に戻った時(`visibilitychange`)に鮮度チェックして古ければ即更新+前面にいる間だけ5分間隔の自動更新(バックグラウンドタブでは `setInterval` がブラウザに間引かれるため、定期更新だけに頼らない)
- 手動更新ボタン(スマホはプルリフレッシュ)
- 自分の操作(タスク完了など)は即時反映
- **自動更新はデータの差分だけを静かに反映**し、スクロール位置・展開状態・入力内容を保持する。ユーザーが編集/閲覧中の項目は更新対象から除外(「アプリが勝手に動く」体験を防ぐ)
- 最終更新時刻は「◯分前に更新」の相対表記。15分以上更新が滞ったら色を変えて警告
- 認証切れ(401)検知時はポーリングを止めて「再接続」表示に切り替える

## 技術実装メモ(実装時の落とし穴)

- **OAuth スコープ(権限範囲)は最小構成**:
  - カレンダー: `calendar.events`(予定の作成・読取・更新・削除)+ カレンダー一覧取得用の readonly スコープ(粒度は実装時に要検証)。全権の `auth/calendar` は避ける
  - Tasks: `tasks`(これ1つで全部)
  - Gmail: `gmail.modify`(読取+既読化+アーカイブ+ゴミ箱移動を全てカバー)
- **繰り返し予定**: 取得は `events.list` に `singleEvents=true`+`timeMin/timeMax`(各回に展開済みで取得)。「この回だけ」の変更・削除は**インスタンス id への PATCH/DELETE**(Google 側が例外レコードを自動生成。親イベントを PATCH すると全回が変わる事故になるので注意)。`recurringEventId` の有無で繰り返しかを判定し編集 UI を出し分ける
- **終日予定の `end.date` は排他的**(終了日当日を含まない)— 日付計算バグの定番
- **Tasks の期限 `due` は日付のみで時刻を持たず、常に UTC 0時で返る**。`new Date()` に変換せず、**先頭10文字(`YYYY-MM-DD`)の文字列として比較・加算**する(タイムゾーン起因の1日ズレ防止)。書き戻しは `T00:00:00.000Z` を付けて PATCH。Google Tasks 側の繰り返しタスクは API から繰り返しルールが見えないため単発として扱う(本家アプリの「完了すると次回生成」挙動に任せる)
- **Tasks はリスト横断の一括取得 API が無く、リストごとに N 回呼ぶ**
- **Gmail 一覧は2段階取得**: `messages.list`(`q=in:inbox is:unread`)は ID しか返さないため、件名・差出人は1通ずつ `messages.get`(`format=metadata`)を並列 fetch(30件なら実用上問題なし)
- **GitHub Pages**: Vite の `base: '/ABOperations/'` を忘れると全アセット404。GIS の「承認済み JavaScript 生成元」には**パスを含まないオリジン**(`https://<ユーザー名>.github.io`)を登録
- **localStorage 設定は単一キー(`abops:settings`)+`{ version: 1, ... }` 形式**。読み込み時にバリデーションし、壊れていたらデフォルト値に静かにフォールバック+バージョン移行関数(設定項目が増える前提のため最初から)
- **PWA は最小構成**: manifest+アイコン+`apple-touch-icon` のみ。Service Worker(オフライン用の仕組み)は初期スコープ外
- レート制限(API 呼び出し上限)は個人1人×4端末×5分間隔なら無料枠に余裕で収まる
