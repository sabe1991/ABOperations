# AB Operations

Google カレンダー・Google Tasks・Gmail に天気・ニュースを加えて **1画面に集約した自分専用ダッシュボード**（司令塔）です。
複数のアプリを行き来せずに、その日の予定・タスク・メールをまとめて把握することを目的としています。

公開URL: https://sabe1991.github.io/ABOperations/

---

## 何を解決するツールか

- **「タスクが増えても予定が埋もれない」分離表示**
  Google カレンダーではタスクと予定が同じ場所に出るため、タスクが多いと予定が見えなくなります。本ツールは予定・タスク・メールを別々のパネル（画面上の区画）に分けて表示します。
- **「1つのURLを開けば全部済む」司令塔化**
  複数アプリを見に行く手間・見落としをなくします。
- 将来ここに機能を統合していく **土台** とする。

## 主な機能

| パネル | できること |
|---|---|
| 今日の予定（24hタイムライン） | 今日の予定を時間軸に配置（広い画面でのみ表示） |
| 今後の予定 | 今日から先の予定を日付ごとに表示。予定の**作成・編集・削除**（単発／繰り返しの「この回のみ」）に対応 |
| カレンダー（月ミニカレンダー） | 5週間ぶんの日付を表示。予定のある日にドット、日付クリックで予定パネルへスクロール（広い画面でのみ表示） |
| タスク | 全リストの未完了タスクを「期限切れ／今日／明日／以降／期限なし」に分類。**完了・追加・期限変更・編集・削除**に対応（誤操作は「元に戻す」で取り消し可能） |
| 天気 | 現在＋3日ぶんの予報（Open-Meteo。地点は設定で変更可・広い画面でのみ表示） |
| メール（Gmail） | 受信トレイの未読／既読を一覧表示。本文プレビュー（埋め込み画像・添付ファイルのダウンロードに対応）と、**既読化・未読に戻す・アーカイブ・スター・新規作成／返信の送信**（端末ごとにオン/オフ。既定はオフ） |
| ニュース | **メールをオフにしている端末で、メールの代わりに出る枠**。Qiita / Wikipedia 注目 / 地震情報 / Hacker News / dev.to / GitHub / 宇宙ニュースから**最大3つを設定で選び**、タブで切り替えて一覧表示（クリックで別タブに開く） |

- 画面が狭い端末（スマホ）は下端のタブで「予定／タスク／メール」を1枚ずつ切り替えます。タブには未読メール数・期限切れタスク数のバッジが出ます。
- 配色テーマ（明るさ）は OS のライト/ダークに自動追従し、設定から手動固定も可能。
- 設定（歯車アイコン）はモーダル（画面に重ねて出す小窓）で表示。ページ遷移は行いません。設定できるのは、配色テーマ／メール（Gmail）の表示／ニュースソースの選択／予定・タスクの出典名の表示／メールの外部画像の表示／週の開始曜日／天気の地点です。

### キーボードショートカット（PC）

| キー | 動作 |
|---|---|
| `R` | すべてのデータを更新 |
| `/` | タスクのクイック追加へ移動 |
| `Esc` | 開いているモーダル・シートを閉じる |

入力中は無効です。スマホでは画面右上の更新ボタン（↻）を使います。

## 対応デバイス・使い方

自宅 Mac / 会社 Windows / Android / iPhone — **すべてブラウザで同じ URL を開くだけ**です。

- スマホは「ホーム画面に追加」でアプリ風に起動できます（PWA＝ブラウザのページをアプリのように扱う仕組み）。
- 新しい端末では初回だけ Google ログイン（認証）が必要です。
- **設定は端末ごと**に保存されます（ブラウザの `localStorage`＝ブラウザ内の保存領域。端末間で同期はしません）。「会社PCだけ Gmail をオフ」のように端末別の使い分けができます。
- ⚠ iOS（iPhone）のホーム画面起動では Google ログインが動かない事例が報告されています。動かない場合は iPhone のみ Safari のブックマーク運用としてください（Android は問題なし）。

## アーキテクチャ（全体の作り）

| 項目 | 決定 |
|---|---|
| 方式 | **サーバーを持たない静的Webアプリ**。各端末のブラウザが Google API から直接データを取得する |
| ホスティング | GitHub Pages（リポジトリ名 `ABOperations`）。コードは公開されるが秘密情報は含めない（API キーは作らない・置かない。トークン付きの通信で直接呼べるため不要） |
| 技術 | React + Vite + TypeScript。`main` ブランチへの push で GitHub Actions が自動ビルド＆デプロイ |
| 画面遷移 | **単一画面**のため URL によるページ切り替え（ルーティング）は導入しない（設定はモーダル表示） |
| 認証 | Google Identity Services（クライアントサイドの OAuth）。Google Cloud プロジェクトは**テストモード**で自分専用に運用 |
| データ取得 | TanStack Query（取得結果のキャッシュ・再取得を管理するライブラリ）。前面復帰時の更新を主・5分間隔のポーリングを従とする |
| 外部API（天気・ニュース） | **APIキー不要・CORS 対応・JSON を直取りできる公開APIだけ**を使う。サーバーを持たない構成のまま、鍵の管理もせずに済ませるため（該当しないソースは採用しない） |

用語補足:
- **OAuth（オーオース）**: パスワードをアプリに渡さず、Google 側で「このアプリに何を許可するか」だけを与える認証の仕組み。
- **ポーリング**: 一定間隔でデータを取りに行って最新化すること。

### 段階的認可（会社PCへの配慮）

権限（スコープ）は必要なものだけを順に求める **段階的認可（incremental authorization）** を採用しています。

- 初回同意は **カレンダー＋Tasks のみ**。
- **Gmail の権限は、Gmail パネルを有効化した端末でだけ追加同意**します。会社PCのトークンは最初から Gmail 権限を持ちません。

### 再ログインの頻度（重要）

Google の仕様上、次のタイミングで再ログインが必要になります（設計上の既知の制約）:

- **アクセストークン（API を呼ぶための一時的な通行証）は約1時間で失効**します。開きっぱなしで使う場合、約1時間ごとに「再接続」ボタンを1クリックします。
- **テストモードでは同意自体が7日で失効**します（Google 公式仕様）。各端末で週1回、同意画面をくぐり直します。
- 認証切れ（HTTP 401）を検知したら、パネルの表示データは残したまま、上部に「再接続」を**1本だけ**出す UX にしています。
- アクセストークンは **`sessionStorage`（同じタブが開いている間だけ残る一時保存領域）に保存**します。ページ更新（リロード）ではログインが維持され、**タブを閉じると消えます**。`localStorage`（ブラウザを再起動しても残る保存領域）は使わないので、端末に長期保存はしません。どのみち約1時間で失効するため、それより短い期限で古いトークンは破棄します。
  - トレードオフ: `sessionStorage` は XSS（悪意あるスクリプト実行）で読み出され得る点がメモリ変数のみの保持と異なります。それでも採用したのは、リロードのたびに再ログインを強いる不便を避けるためです。多層の本文サニタイズ（下記）で XSS の入口自体を塞ぐことを主防御とし、保存範囲は「同じタブの間だけ」に限定してリスクを抑えています。

## セキュリティ方針

メールは外部から任意の HTML（Webページを記述する言語）が送り込まれる唯一の入口のため、本文表示を多層で防御しています。

- HTML 本文は **DOMPurify（定番の無害化ライブラリ）で浄化**したうえ、**`sandbox` 属性付きの `<iframe>`（隔離された枠）**に流し込んで本体から切り離して表示。
- iframe 内に CSP（Content Security Policy＝読み込み先を制限する指定）を埋め、`default-src 'none'` でスクリプト・通信をすべて遮断。読み込みを許すのは画像・インラインスタイル・`data:` のフォントだけです。
- **リモート画像（送信者のサーバーから取りに行く画像）の扱いは設定「メールの外部画像を表示」で切り替えます（既定はON＝表示）。** 表示すると開封（時刻・IP＝おおよその所在地）が送信者に伝わりうるため、OFF にすると CSP の `img-src` を `data:` だけに絞ってブロックし、本文の「画像を表示」で1通ずつ解禁できます。既定ONのハードニングとして、画像取得時に参照元 URL を送らない `<meta name="referrer" content="no-referrer">` を本文 iframe に入れています。
- メールに同梱された**埋め込み画像（`cid:` 参照）は外部通信を伴わない**ため、上記の設定に関わらず `data:` URI（画像データを本文に埋め込んだ形式）に変換して常に表示します（署名のロゴなどが見えるようにするため）。
- メール内の生の URL は、クリックできるリンクに変換したうえで `rel="noopener noreferrer"` を付与。

## 操作の確認方針（破壊的操作）

「削除」などのやり直しにくい操作について、いつ確認ダイアログ（"本当に実行しますか？" の問い合わせ）を出すかの基準を統一しています。

- **やり直せる操作（Undo 可能）＝確認を出さない。** 予定・タスクの削除、メールの既読化・アーカイブなどは、直後に「元に戻す」スナックバー（画面下部に一定時間出る取り消しバー）を表示します。取り消せる安全網があるので、毎回ダイアログで手を止めさせない方が快適だからです。
- **やり直せない操作（不可逆）＝確認を出す。** ログアウトは、保持しているトークン（通行証）と接続状態を破棄し「元に戻す」がないため、`window.confirm`（ブラウザ標準の確認ダイアログ）で一度確認します。
- 新しく破壊的な操作を足すときは、まず「Undo を用意できるか」を検討し、用意できるなら無確認＋Undo、できないなら確認ダイアログ、のどちらかに寄せます。

## 開発

前提: Node.js 22 系（GitHub Actions もこのバージョンでビルドします）。

```bash
npm install       # 依存パッケージのインストール
npm run dev       # ローカル開発サーバー（Vite）を起動
npm run build     # 型チェック（tsc）＋本番ビルド（vite build）
npm run preview   # ビルド結果をローカルで確認
npm run format    # Prettier で整形
```

### デプロイ

`main` ブランチへ push すると、GitHub Actions（`.github/workflows/deploy.yml`）が自動でビルドし GitHub Pages へ公開します。手動デプロイは不要です。

- Vite の `base: '/ABOperations/'` を変えるとアセットのパスが崩れるので注意（全アセット404の原因になります）。
- GIS の「承認済み JavaScript 生成元」には**パスを含まないオリジン**（`https://sabe1991.github.io`）を登録します。

```
src/
  App.tsx                 画面全体の骨格・パネル配置・レイアウト分岐
  main.tsx                エントリーポイント（テーマ初期化・描画開始）
  index.css               デザイントークン（色・書体・角丸・影）と全スタイル
  config.ts               OAuth クライアントID とスコープ定義（段階的認可のグループ分け）
  Skeleton.tsx            読み込み中のプレースホルダー表示
  ErrorBoundary.tsx       パネル単位のエラー表示（取得エラー・描画時例外を同じ見た目で扱う）
  PanelLink.tsx           見出し右上の「外部サービスを開く」リンクボタン
  queryClient.ts          TanStack Query の設定（401 の共通ハンドリング等）
  errorMessage.ts         例外をユーザー向けの短い日本語メッセージへ変換
  fetchTimeout.ts         認証不要の外部API向けタイムアウト付き fetch
  pwaUpdate.ts            PWA の更新検知と適用（更新を促すバナー）
  roving.ts               タブリストの矢印キー操作（roving tabindex）
  tempId.ts               楽観更新の仮エントリ用の一時ID
  useDialog.ts            モーダル/シート共通の Esc・フォーカストラップ・初期フォーカス
  useLastUpdated.ts       ヘッダーに出す「データの最終更新時刻」
  useMediaQuery.ts        画面幅に応じた密度型レイアウトの判定
  auth/                   認証（GIS ラッパー・トークン保持・接続状態ストア・要求スコープの決定）
  google/
    fetchJson.ts          Google API 呼び出しの共通処理（401/403 検知など）
    pool.ts               N+1 取得の同時実行数制限（1件の失敗で全滅させない）
  features/
    calendar/             予定・24hタイムライン・月カレンダー
    tasks/                タスク
    gmail/                メール（一覧・本文の安全な描画・既読/アーカイブ/スター/送信）
    news/                 ニュース（鍵なし公開APIの一覧・ソース切替）
    weather/              天気（Open-Meteo）
    settings/             設定モーダル・表示設定・アカウント情報
```

各パネルは独立したコンポーネント（部品）として実装しており、将来のレイアウト変更・機能追加の土台になっています。

## 初回セットアップ（新しく自分用に立ち上げる場合）

1. Google Cloud で無料プロジェクトを作成し、Calendar / Tasks / Gmail API を有効化、OAuth クライアントID を発行する（目安30〜60分）。
2. OAuth 同意画面はテストモードのまま運用し、自分をテストユーザーに登録する。
3. 各端末の初回アクセス時に Google 許可画面で「続行」（未確認アプリの警告が出るが自作のためOK）。
