## Changed

- Android の PWA 本体でメール本文中のリンクを押したとき、アプリ内の Chrome(Custom Tab)ではなく Chrome 本体で開くようにした(ユーザー要望)。スタンドアロン起動の PWA では外部リンクが既定で Custom Tab に流れるため、Android かつスタンドアロン起動のときだけ(`ANDROID_STANDALONE` 判定)、リンクの遷移先を Android の intent URI(`intent://…;scheme=https;action=android.intent.action.VIEW;end`、`toIntentUrl`)に変換して Chrome 本体を起動する(best-effort)。iframe 内から直接 intent を発行すると sandbox にブロックされうるため、`allow-same-origin` を利用して本文 iframe のリンククリックを隔離枠の外(非sandbox の本体側)で横取りし、本体側に一時的な `<a>` を作って intent を発行する方式にした(PWA 自体は開いたまま)。Mac や Android の通常タブでは従来どおり `target="_blank"` で新規タブに開く。URL のフラグメント(#以降)は intent 構文と衝突するため変換時に落とす(メールでの利用は稀)。`mailto:` 等 http/https 以外は変換せず既定動作に任せる (2026-07-12)
- カレンダー予定の作成・編集フォームに「場所」と「メモ(詳細)」の入力欄を追加した(フェーズ4-c の仕上げ)。予定の `location`/`description` を読み書きできるようにし、一覧では場所があれば予定行に「📍場所」を小さく表示する。patch では空文字を送れば場所・メモを消去できる。時刻に加えて場所・詳細も登録できるようになった (2026-07-12)
- アクセストークンの保存先をメモリのみから **sessionStorage** に変更した(ユーザー判断)。デスクトップ(Mac/Windows)のブラウザで、ページ更新や再訪問のたびに毎回ログインが必要だった問題への対処。起動時に sessionStorage の有効なトークン(TTL 55分)を復元して即接続する経路(`restoreSession`)を主とし、無ければ従来のサイレント認証→明示ログインの順にフォールバックする。sessionStorage はタブを閉じると消えるため「端末に長期保存しない(localStorage を避ける)」方針は概ね維持。デスクトップでサイレント認証(`prompt:''`)がサードパーティCookie制限で機能しなかったのが背景(Android アプリ版はプロセス保持でトークンが生き残るため元々再ログイン不要だった) (2026-07-12)
- `PLAN.md` にプロUI/UXデザイナー視点+シニアエンジニア視点のレビュー結果を反映。(1)誤操作対策(完了・アーカイブ・削除の Undo、操作の3層配置)、(2)スマホタブのバッジと初期タブ、(3)例外状態(未ログイン/空/エラー/読み込み中)の設計、(4)再ログイン頻度の現実的な見積もりへの修正と段階的認可の採用、(5)メール本文表示のセキュリティ対策(サニタイズ+隔離枠+画像ブロック)、(6)API実装の落とし穴メモ、(7)縦切り実装の段取り、などを追記。`TODO.md` に #6〜#8 を追加 (2026-07-12)

- Android の PWA でメール本文のリンクがまだアプリ内(Custom Tab)で開く問題への対処。(1)適用条件を「Android かつスタンドアロン起動」から「Android なら適用」に広げた。WebAPK では `display-mode: standalone` が期待どおり返らずリンク横取り自体が発動していなかったため(`ANDROID_STANDALONE`→`IS_ANDROID`)。(2)intent URI に `package=com.android.chrome`(開き先を Chrome 本体に固定=Custom Tab 回避の要)と `S.browser_fallback_url`(未解決時のフォールバック)を追加。(3)切り分け用に、Android のときだけ本文上に「リンクは Chrome で開きます」の目印を一時表示(判定が効いているか実機で確認するため。動作確定後に外す) (2026-07-12)

- 権限不足（403 insufficient_scope）を認証切れ（401）と区別して扱うようにし、タスク機能を追加する前の「カレンダーだけの許可」で使い続けている端末（主に Android アプリ版。プロセスが生き続けて再ログインが起きないため古い許可のまま更新されない）で「タスクの取得に失敗しました」と一般エラーになっていたのを修正。(1)`fetchJson` が 403 の `WWW-Authenticate: insufficient_scope`／本文の `ACCESS_TOKEN_SCOPE_INSUFFICIENT` を検出して専用の `ScopeError` を投げる。(2)`authStore` に `needsScope` 状態と `markNeedsScope()` を追加し、`QueryCache`/`MutationCache` の onError で `ScopeError` を拾って追加同意を促す状態に切り替える。(3)ログイン成功時にも、必要なスコープ(`INITIAL_SCOPES`)に不足があれば API 失敗を待たず先回りで `needsScope` を立てる。(4)画面上部に「タスクを表示するには追加の許可が必要です」バナー＋「許可する」ボタンを出し、押すと段階的認可で不足スコープだけを追加同意する（再ログイン不要）。同意後は `needsScope` が解けてタスク取得が自動復活。(5)権限不足中はタスク取得を止め(`enabled` に `!needsScope`)403 の叩き続けを防止。(6)復元経路でも起動時に必ず GIS を先読みし、「許可する」クリックで同期的にトークン要求できるようにした (2026-07-12)
