## 優先度: 中

- **#15** Android の PWA(WebAPK)で更新の反映が遅い問題。`vite-plugin-pwa` の `registerType: 'autoUpdate'` を使っているが、Android の WebAPK はプロセスを保持するため、デプロイ後に開き直しても古いキャッシュ(Service Worker がプリキャッシュした旧 JS)が動き続けることがある(実際、リンク intent 対応の診断表示が Android で全く出なかった＝新コードが動いていなかった)。Mac のブラウザでは概ね即反映される。対策案: (1)新バージョン検知時に「更新があります・再読込」トーストを出す(`vite-plugin-pwa` の `registerType: 'prompt'` + 明示リロード)、(2)アプリ起動/フォアグラウンド復帰時に SW の更新チェック(`registration.update()`)を明示的に叩く、(3)画面のどこかに常時ビルド版(コミットハッシュ)を小さく表示して、実機でどの版が動いているか一目で分かるようにする。動作確認は基本 Mac で行うため優先度は中。
- **#2** レイアウト切替機能。初期バージョンは PC=3カラム固定・スマホ=タブ固定。3カラム型を数週間使ってから「今日フォーカス型」などの第2レイアウトの要否を判断し、設定画面から切り替えられるようにする。

## 優先度: 低

- **#3** 繰り返し予定のルール自体の編集(毎週→隔週など)への対応。初期バージョンでは Google カレンダー本家で行う運用。頻度が低いため UI の複雑さに見合うか使いながら判断する。
- **#4** Gmail の返信・新規作成機能。現状 Gmail を連絡用途に使っていないため未実装。必要になったら追加を検討する。
- **#5** 設定の端末間同期。現状は各端末の localStorage に個別保存(会社PCだけ Gmail オフにする等、端末別が都合良い面もある)。設定項目が増えて手動設定が煩雑になったら、Google Drive の隠し領域(appDataFolder)への保存を検討する。
- **#6** オフライン対応(データのキャッシュ)。Android の PWA インストール(WebAPK 化)に Service Worker が事実上必要だったため、`vite-plugin-pwa` による最小 Service Worker を導入済み(アプリシェル=HTML/JS/CSS のみプリキャッシュ)。これにより「アプリの枠は電波が無くても開く」ようになったが、**取得データ(予定・タスク・メール)のオフラインキャッシュは未対応**。移動中の閲覧ニーズが強ければ「最後に取得したデータのキャッシュ表示+『オフラインです』バナー」を追加検討する(書き込み操作のオフライン対応はしない)。
- **#7** PC 版のキーボードショートカット(R=更新、/=クイック追加 など)。初期バージョンは「Tab で移動でき、フォーカスが見える」最低限のみ。
- **#10** 複数日にまたがる予定の作成・編集UI。フェーズ4-c の予定作成/編集フォームは「単日」前提(日付フィールド1つ+開始/終了時刻)。既存の複数日予定を編集しても日数スパンは保持するが、新規に複数日予定を作る導線は無い。日跨ぎ・複数日の予定を作りたくなったら、フォームに終了日フィールドを追加する。頻度が低い想定のため後回し。
- **#9** タスクの時刻(時分)非対応の周知。Google Tasks 公開API(v1)は期限が日付のみで、時刻は読み書きできない(純正アプリで付けた時刻も公開APIには出ず、`updateTask` で期限を書き換えると日付のみのタスクに変わる)。ユーザーが「時刻付きタスク」を期待して混乱しうるため、(1)編集ボトムシートに「タスクは日付単位です。時刻が必要な予定はカレンダーへ」の一言を添える、(2)時刻付きにしたいときはカレンダー予定作成へ誘導する導線、などを検討。時刻が重要な項目はカレンダー予定で扱う、という設計の使い分けを UI で伝える。
- **#11** メール本文のインライン画像(`cid:` 参照)の表示。本文プレビューは HTML をサニタイズ+sandbox iframe で表示するが、メール内に埋め込まれた画像(`multipart/related` の `cid:` 参照)は表示せずプレースホルダに落としている(真面目に対応すると `attachments.get` で取得→`data:` URI 化→`cid:` 置換が必要でスライスが倍増するため割り切り)。埋め込み画像を見たいケースが増えたら対応する。
- **#12** メール本文の文字コード対応。本文デコードは UTF-8 決め打ち。日本語メールで多い `ISO-2022-JP`(いわゆるJISコード)等は化ける可能性がある。化けるメールが実際に出たら、パートの `Content-Type` の charset を見て `TextDecoder` の指定を切り替える対応を足す。
- **#14** Android でメール本文のリンクを Chrome 本体で開く(intent 方式)の実機検証。`IS_ANDROID` のときリンククリックを本文 iframe の外(本体側)で横取りし、`intent://…;package=com.android.chrome;S.browser_fallback_url=…;end` で Chrome 本体を起動する実装は入れてあるが、Android 実機で新コードが動いていなかった(#15 のキャッシュ問題)ため未検証。#15 が解消して新コードが動く状態で、(1)リンクが Custom Tab でなく Chrome 本体で開くか、(2)開かない場合は intent 起動側の調整(package 指定の要否、`window.location.href` 方式への変更、TWA 化の要否)を検討する。Mac/通常タブでは従来どおり `target="_blank"` で問題なし。
- **#13** メール添付ファイルの表示・ダウンロード。本文プレビューは本文テキスト/HTMLのみで、添付ファイル(PDF・画像等)は一覧も表示していない。添付を確認したいニーズが出たら、ヘッダのファイル名一覧表示→`attachments.get` でのダウンロードを検討する。
- **#8** 再ログイン頻度の対策。テストモード運用では「約1時間ごとに再接続1クリック+各端末で週1回フル同意」が発生する(`PLAN.md` 参照)。なお、ページ更新でメモリが消えて毎回ログインになる問題は sessionStorage 採用で解消済み(同タブ内は維持)だが、1時間のトークン失効・7日の同意失効・新規タブ/タブ閉じ後の再ログインは残る。第1週の運用で苦痛と判明したら、(1) Cloudflare Workers 等の無料枠で動く極小の中継サーバー(トークン交換のみ)を追加して認可コードフローに移行、(2) アプリの本番公開(Gmail の制限付きスコープは審査が必要な点に注意)、のいずれかを検討する。
